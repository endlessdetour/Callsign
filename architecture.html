Architecture

How Callsign is structured — three planes that keep control, data, and the client cleanly separated.

Design goals

  • Expose only HTTPS on port 443 externally — everything else is closed.
  • A driver-oriented client data path (Wintun on Windows).
  • Validate transport and control flow first, then harden.

The three planes

1 · Control plane

A Flask app (behind gunicorn + nginx) that bootstraps a device, issues a session token, and handles heartbeat and token validation. It also serves the admin console at /login.

2 · Data plane

WebSockets over WSS. It accepts an authenticated tunnel connection and carries binary frames over TLS (terminated at the reverse proxy). Per-device IP leases avoid address collisions.

3 · Client plane

The Windows agent performs bootstrap, keeps a heartbeat loop, connects the tunnel, and exchanges frames through a tun adapter. An adapter abstraction lets the mock adapter be swapped for the real Wintun driver.

Request flow

Client  ──HTTPS──▶  nginx :443  ──▶  Control plane (Flask)   bootstrap / token / heartbeat
Client  ──WSS────▶  nginx :443  ──▶  Tunnel plane (WebSocket)  authenticated binary frames
                                  ▲
                                  └─ everything else → 444 (closed)

Production direction

  • Nginx in front for TLS termination and 443-only exposure.
  • Move from a single shared access token toward mTLS + signed short-lived tokens.
  • Replace the mock adapter with Wintun plus route management.
  • Replace the echo tunnel with real packet forwarding / NAT.
